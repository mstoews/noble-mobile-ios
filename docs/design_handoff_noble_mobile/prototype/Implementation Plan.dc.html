Noble Ledger Mobile · Design → Build

Implementation Plan

Bringing the reviewed prototype into the shipping SwiftUI app (nbledger) — brand alignment, a restructured information architecture, and the new screens, delivered in reviewable phases.

Target iOS 17+ · SwiftUI · Swift 6 Source of truth Noble Ledger Prototype.dc.html Design system NobleLedger (emerald / slate) Owner Murray Toews

Goal. Reskin and re-architect the existing app to the NobleLedger brand and the 2-tab + Capture companion model, without rewriting the data layer. APIService, all Codable models, auth, and Plaid integration stay as-is; this is a presentation-and-navigation effort that reuses the current networking untouched.

01 Guiding principles

  • Reskin, don't rebuild. Every screen already has a working data source. We change layout, hierarchy, and color — not the fetch/decode logic.
  • One design-token layer. All color/spacing/type flows from a single Swift tokens file mirroring colors_and_type.css. No ad-hoc hex values in views (the current navy Color(red:0.14…) literals get deleted).
  • Ship behind phases, not a big bang. Each phase below is independently mergeable and demoable; navigation flips last so half-migrated screens never ship.
  • Native components first. Prefer TabView, NavigationStack, List, .searchable, SF Symbols and system materials over custom chrome, so accessibility and dark mode come mostly free.

02 Phase 0 — Design foundations

FOUNDATION

Tokens, type, brand assets

~2–3 days
  • Create Theme/NobleTokens.swift: emerald ramp (primary #047857, highlight #065f46), slate neutrals, semantic roles (success/warn/overdue), radii (10/16), and the money/tabular number style. Retire the two navy gradient literals in MainView, MoreView, LoginView.
  • Add the crown logo asset (framed-container lockup) and set AccentColor to emerald-700.
  • Bundle Inter var (already the DS face) + register; adopt tabular figures for all currency via a shared Text.money() modifier.
  • Build a small SwiftUI kit: NobleCard, SectionLabel, StatusPill, MetricHero, DetailRow — the repeated primitives the prototype uses on every screen.

Exit: a Storybook-style #Preview gallery renders every primitive in light + dark.

03 Phase 1 — Navigation & IA

STRUCTURE

2-tab companion + Capture FAB + More hub

~3–4 days
  • Rework MainView.swift from the 5-tab TabView to Home · Activity · [Capture FAB] · More. Keep the FAB overlay pattern already present; wire it to the capture flow instead of chat.
  • Promote the daily jobs: Payment Sign-Off and Journal Booking become the top "Needs your action" group in MoreView, with live counts pulled from the existing journal/payment fetches.
  • Fold overlapping concepts: "Invoices" → Capture, one Journals entry (list + booking), Payables surfaced once. Route the rest (Payables, Receivables, Banking, Accounts, Settings) as NavigationLink pushes from More.
  • Relocate the AI assistant to a More entry / toolbar item so the FAB no longer collides with tab bar and list content.

Exit: every destination reachable; badge counts bind to real data.

05 Phase 3 — Login & capture loop

FLOWS

Branded login + AI capture → draft payable

~4–5 days
  • Login (LoginView.swift): emerald/slate gradient, framed crown, remembered workspace (persist tenant in @AppStorage / Keychain), Face ID first. Company ID stops gating Sign in with Apple. Error copy stays declarative per DS voice.
  • Capture loop (InvoicesView.swift): the existing VisionKit scan → /analyze_invoice → confirm flow now lands the created bill in Payables with a success banner and a DRAFT state, closing the loop the prototype demonstrates.
  • Wire the primary actions — Sign off, Book, Record payment — to mutate local state (clear item, decrement badge) on success, behind the existing confirmation-dialog + Face ID pattern.

06 Phase 4 — Assistant safety & states

SAFETY

AI assistant + empty/error states

~3 days
  • Rework AgentChatView empty state: questions as suggestions; move "Book / Close all open journals" into a Bulk actions · confirmation required group with counts, an explicit review step, and Face ID — never one-tap.
  • Design empty ("All caught up"), loading, and error/offline states for each list, replacing the current bare ProgressView text.

07 Cross-cutting workstreams

Dark mode

Every primitive ships light + dark from Phase 0; audit each screen against the DS slate-900 surfaces. Honor the Settings toggle + system setting.

Accessibility

Dynamic Type on all text, 44pt hit targets, VoiceOver labels on icon-only controls and status pills (the code already labels evidence counts — extend the pattern).

Motion

Lean on .navigationTransition / view transitions; keep it restrained per brand (no scale-on-press).

QA & snapshots

Extend the existing UITest screenshot suite (AccountsPageScreenshotTests, ApprovalScreensScreenshotTests) to cover new screens in light + dark.

08 Sequencing, risk & done

Suggested order

Phase 0 → 1 → 2 (top-down by table) → 3 → 4. Navigation (Phase 1) merges but stays feature-flagged until enough Phase 2 screens are reskinned to avoid a half-branded shell.

Risks & mitigations

  • Signed balances render red. The dashboard shows raw signed values (liabilities look like errors). Fix in the presentation layer with display conventions, not by touching the ledger math.
  • Concurrency. Honor the existing rule — sequential API calls, avoid async let with mixed error handling — when refactoring dashboard aggregation.
  • Scope creep on the FAB. Capture is the hero; resist adding a second primary action to the tab bar.

Definition of done (per screen)

  • Matches the prototype in layout, hierarchy, and DS tokens; zero hardcoded colors.
  • Renders correctly in light + dark, at largest Dynamic Type, with VoiceOver.
  • Loading / empty / error states designed; primary action wired to real data with confirmation where destructive.
  • Snapshot test added and green.
